Testing is one of the ways Grails makes web development fun. The framework provides all the tooling you need to quickly build a rich test suite for your web and API applications.

When debugging a problem my first or second step is to isolate the problem into a unit or integration test. This helps me reason about the problem as well as provide an example for opening a bug or asking a question.

In the book testing is introduced early and often. All of the sample code is provided with tests directly into the book.

Testing

We’ve already seen a few examples of Grails unit testing as we’ve gone through concepts. We’ve been using the Spock testing framework. In this section we’ll take a deeper dive into each type of testing and Grails makes testing easy to test your application.

Lets take a look at the testing framework and how we can use the provided tools to make writing our test easy.

Spock

Grails includes the Spock as it’s default testing framework. Spock is a general testing framework can test any Java or Groovy class, not just Grails.

The first thing you’ll notice is the test method names are strings. This makes it easy to write very descriptive and readable test cases.

Inside your test you’ll see block such as when:, then: with an optional description as well. These blocks organize how your test is executed.

Grails Test

Lets look at some of the testing facilities Grails adds into Spock to allow you to easily test.

@TestFor

The @TestFor(ArtifactClass) annotation examines the Artifact passed in and sets up your test for that type of Artifact. This works on Controllers, Services, Domain classes & TagLibs.

With this annotation applied Grails will provide the appropriate objects into our class for us and any related objects we may need following this convention:

  • Controller → controller

  • Service → service

  • Domain → domain

  • TagLib → tagLib

Controllers and TagLibs also have a request and response objects, and Domain classes and Services do not. If you find yourself needing an web request in a service or domain class you may want to evaluate how your logic is working to avoid this.

Specification

All tests extend the Spock Specification class. This is what makes your test a Spock test.

HibernateSpec

There are other Specifications Grails provides such as HiberanteSpec. The HibernateSpec will setup all the internals needed to test classes with Transactions or other database interactions.

In tests where you do interactions with Hibernate, transactions, and sessions set your test up like this:

@TestFor(ExampleService)
class TransactionTestsSpec extends HibernateSpec { }

You can now write your unit tests that interact with transactions and sessions as normal.

This is still not a true Hibernate/Database environment and there may be slight difference. Any test here you should consider including as an Integration test.

Unit

By far the easiest way to add testing to your app is by filling in the unit tests Grails creates for you when you create an artifact. Unit tests are stored by convention under src/test/groovy/. Grails will create a test named ArtifactSpec.groovy for you, though you can create more Spec files if they get too long, for example: InvoiceRefundSpec.groovy & InvoiceChargeSpec.groovy. This also helps you organize your tests.

When writing unit tests, it is often too easy to stat testing more than you should. If you find yourself having to mock out multiple objects and services it might be a good idea to review how complex your method is.

A unit tests should strive to:

  • Test one thing, not test dependent services

  • Test the positive and the negative case.

  • Not rely on external data (databases, web-services)

  • Tests should be independent of each other.

  • Never trust a test you’ve not seen fail.

These are the basic tenet’s we’ll follow when working through this section.

Integration

Up until now we’ve been writing unit tests for our application. Good unit coverage is essential for testing and validating changes to your app, but you wont know if it all works together unless you also have good integration tests.

The biggest difference between integration and unit testing is integration test actually start up your Grails application and connect to real environment infrastructure such as databases which may behave slightly differently in practice than then in memory databases used for unit tests.

Though they are quite different, many of the same tenets apply to integration

  • Test one interaction per test.

  • Keep your tests data clean and avoid pollution

  • Try to avoid dependancies between test suites

To create an integration test suite we can use the Grails CLI and they will be stored in the aptly named src/integration-test directory:

$ grails create-integration-test InvoiceTransfer
| Created src/integration-test/groovy/intro/concepts/InvoiceTransferSpec.groovy

By default these tests will have two annotations:

  • @Integration which will start up your entire application, run it on a randomly assigned port. This is set as a property serverPort on your test class incase you need it for testing. Running on a random port helps when in CI environments.

  • @Rollback will automatically roll back any database interactions in the test. This helps prevent test data pollution. Note any data setup done in the setup() method will be persisted outside the transaction and will not be rolled back (leading to test pollution)

  • @DirtysContext (optional) will cause the app to stop and start again for the test. Useful in some cases but will significantly increase test run time.

Domain classes

To test a domain class you would interact with it the same way you would interact with it in a controller or a service. This is because your application is actually running the code!

Lets start by validating our domain class works and saves:

src/integration-test/groovy/intro/concepts/AccountDomainSpec.groovy
@Integration
@Rollback
class AccountDomainSpec extends Specification {
    void "An account with name and balance saves"() {
        expect:
        Account.count == 0 (1)
        new Account(name:"Savings",balance: 1_00).save() (2)
        Account.count == 1 (3)
    }

    void "The account created previously is rolled back"() {
        expect:
        Account.count == 0 (4)
    }
}
1 When starting our app, there should be nothing in the database.
2 We’ll create a new valid domain and save it. Remember save() returns true so this assertion will pass.
3 Now there will be 1 Account in the database.
4 Since this class is annotated with @Rollback any persisted data is rolled back and will not pollute other tests.

This scenario illustrates how to use a Domain class in a integration test. For simple scenarios such as this a unit test would suffice, and execute much more quickly.

Services

A well designed app will have most of it’s data access in services. To test a service in an Integration test you use the @Autowired annotation to inject it. Lets look at an example:

src/integration-test/groovy/intro/concepts/InvoiceTransferSpec.groovy
@Integration
@Rollback
class InvoiceTransferSpec extends Specification {
    @Autowired AccountService accountService (1)

    void "Transfer"() {
        setup:
        def from = new Account(name: "From", balance: 1_00).save()
        def to = new Account(name: "To", balance: 1_00).save()

        when:
        accountService.transfer(from.id, to.id, 50) (2)

        then:
        from.balance == 50
        to.balance   == 1_50

        and:
        Account.count == 2
    }

    void "didn't pollute"() {
        expect:
        Account.count == 0  (3)
    }
1 The @Autowired annotation provides us our Grails service to test with.
2 Since this is the actual service, we can call it the same as we would in our Grails application code.
3 The @Rollback annotation on the test clears the data between each test.

Functional

Functional tests use an external runner, such as a real web browser or API client, to execute endpoints in your application. They are similar to Functional test in that a copy of your application is running.

Geb (Browser)

Geb is a WebDriver based browser automation testing solution. It provides a DSL that is jQuery like to select elements on a page and make assertions on them. Grails provides integration into Geb so most of the configuration (such as url, port, etc) is done automatically.

WebDriver is a technology for apps to directly control native web browsers. They many times require a separate install to work correctly. Of course there are a limitations as you can not use a Windows Edge browser on your Mac or Linux machines.

To keep our example simple, we will use the htmlunit driver. htmlunit allows us to run a very basic html browser to test our pages on. In section 2 we will go over setting up a much more complex scenario.

We will use the Grails CLI to setup a Geb test for us:

$ grails create-functional-test Example
| Rendered template FunctionalSpec.groovy to destination src/integration-test/groovy/intro/concepts/ExampleSpec.groovy

This will look very similar to the FunctionalSpec, but instead it extends GebSpec. This provides us with the Geb DSL.

src/integration-test/groovy/intro/concepts/ExampleSpec.groovy
@Integration
@Rollback (1)
class ExampleSpec extends GebSpec { (2)
    void "Test Example index page"() {
        when:"The example page is loaded"
        go '/example/index' (3)

        then:"The body is correct"
        $("body").text() == "Model: Oliver 123" (4)
    }
}
1 The same annotations and behaviors are here from the Integration Spec.
2 GebSpec is a class that extends the Spock Specification - we get all the benefits of Spock here as well.
3 go is part of the Geb DSL - it will navigate to a URL. There are other methods such as click, hover, etc. Anything you can do with a real web browser.
4 A Jquery like selector language makes it easy to write assertions on the rendered HTML.

Geb & WebDriver are vast topics on their own. My personal experience is that there can be value in these types of browser test, but you must be willing to invest the time and effort to keep them updated.

RestBuilder

If you are testing a JSON or XML API there is no need for a full web browser to test. Instead we can use the RestBuilder client to call our own API endpoints and make assertions on the responses.

Grails does not provide a nice RestSpec like they do for Geb, so lets create our own!

First create a new file named RestSpec.groovy in src/integration-test/groovy/intro/concepts/

We want to use the Spock framework, so lets have this class extend Specification.

To make the API calls we will be using RestBuilder from GORM. Include that dependency in our build.gradle

compile 'org.grails:grails-datastore-rest-client'

Now we can put it all together as an abstract class that we can use it to extend our API tests.

src/integration-test/groovy/intro/concepts/RestSpec.groovy
abstract class RestSpec extends Specification {
    String baseUrl
    RestBuilder rest

    def setup() {
        baseUrl = "http://localhost:${serverPort}" (1)
        rest = new RestBuilder()
    }
}
1 serverPort is provided to our integration specifications by Grails. You may also want to set the baseUrl differently if you run in a more complex environment.

Lets try it out on the /example/save endpoint. It takes a person object and saves it to the database and responds with JSON.

Create a new file for our Rest test src/integration-test/groovy/intro/concepts/PersonRestSpec.groovy

Since we’ll be extending the RestSpec that we created we’ll have access to the rest object to make API calls and all the things we’ve seen previously like Autowired services or accessing domain classes.

src/integration-test/groovy/intro/concepts/RestSpec.groovy
@Integration
@Rollback (1)
class PersonRestSpec extends RestSpec { (2)
    void "Create a person"() { (3)
        setup: "Create a person via GORM"
        new Person(firstName: "Eric", lastName: "Helgeson").save()

        when: "Create a person via the API"
        def response = rest.post("${baseUrl}/example/save") { (4)
            accept('application/json')
            json('{"firstName":"Foo","lastName":"Bar"}')
        }

        then: "API call success"
        response.status == 200
        response.json.firstName == "Foo"

        and: "there is 2 people"
        Person.count == 2
    }
}
1 Add the annotations to tell Grails that this is an Integration test.
2 Extend our RestSpec
3 Since our RestSpec extended Specification this is a Spock test.
4 We have access to the rest object and a properly configured baseUrl our app is running on.

It is simple to extend and customize test specs to meet your needs. You could take this and instead of just providing a rest client, you could authenticate to a OAuth endpoint and store the token in the test as well so you could easily make authenticated API calls.

Running tests

To run every test suite (unit, integration, and functional) the most common way is to run:

$ grails test-app
# Alternatively with gradle
$ ./gradlew test

You can also selectively run certain test and test suites with flags and patterns provided by the test-app command:

# Run all unit tests
$ grails test-app -unit

# Run all integration tests
$ grails test-app -integration
... Starts an instance of your application on a random port ...
Grails application running at http://localhost:54693 in environment: test

# Run test that match a pattern
$ grails test-app intro.concepts.Account*

# Rerun test when a file changes that matches the pattern
$ grails test-app intro.concepts.Account* -continuous

# Test all Service Specs in the intro.concepts package.
$ grails test-app intro.concepts.*ServiceSpec

# Run a specific integration test spec
$ grails test-app -integration InvoiceTransferSpec
When running tests in a CI environment such as Jenkins you should use the wrapper commands ./grailsw or ./gradlew to ensure you’re always using the correct versions of your build tools.

Reports

Both HTML and XML tests are created when running test-app.

The HTML is used when you want to view them interactively. Open build/reports/tests/index.html to view them.

The XML reports are used by CI environments such as Jenkins for reporting tests over time. The XML reports are stored under build/test-results/.
